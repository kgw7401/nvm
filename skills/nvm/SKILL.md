# /nvm — Engineering Thought-Trajectory Generator

Analyzes Claude conversation logs + git history for a given period,
and produces a **thought-trajectory document** capturing the human's problem-definition process and Claude's problem-solving approach.

---

## Execution Steps

### Step 1: Determine Period

- `/nvm` → today (00:00 ~ now)
- `/nvm week` → last 7 days (`date -v-7d +%Y-%m-%d`)
- `/nvm YYYY-MM-DD` → from that date to now

### Step 2: Collect Data

**Never read raw JSONL with the Read tool.** Use the preprocessor (`scripts/extract_session.py`).

```bash
# 1) Index for activity overview (skip if ≤ 3 projects — go straight to --all)
python3 scripts/extract_session.py --index --since {since_date}

# 2) Extract sessions (default: full user messages + Claude truncated to 200 chars)
python3 scripts/extract_session.py --all --since {since_date} --user-full --claude-max 200
# For specific project only:
# python3 scripts/extract_session.py --project {path} --since {since_date} --user-full --claude-max 200
```

### Step 3: Collect Git History

Extract project paths from index `<!-- path: ... -->` comments.

```bash
git -C {project_path} log --since="{since_date}" --oneline --stat
git -C {project_path} remote get-url origin   # for commit links
```

Only fetch detailed diffs when needed: `git -C {path} log --since="{since_date}" -p -- {file}`

### Step 4: Generate Document

Follow the **Generation Principles** below to write the thought-trajectory document.

**The first line must include the `<!-- generated by NVM -->` marker.** The next line is `# Title` (serves as the one-line summary).

```markdown
<!-- generated by NVM -->
# NVM Cards (2026-03-25 ~ 2026-04-01)
```

### Step 5: Save

Save to `.nvm/{date}.md`. If a file with the same date exists, append `-2`, `-3`, etc.
Report the saved path to the user.

### Language

Generate the document in the same language the user used in their conversations.
If conversations are mixed-language, prefer the language used most frequently.

---

## Generation Principles

### 1. Free Narrative + 3 Required Elements

Do not enforce fixed fields (`[Problem]`, `[Decision]`, etc.).
Structure freely based on content type, but **every card must include these 3**:

1. **One-line summary** — card title (`###`). What is this card about at a glance.
2. **Core content** — varies by type:
   - Decision → choice, rejected alternatives, reasoning, trade-offs
   - System learning → key structure, behavior, discoveries
   - Debugging → symptoms, elimination process, **specific fix** (which file, which code, how it changed)
   - Org/process → context, key concerns, decisions made
   - **Never abbreviate the resolution process.** Instead of "modified 3 places", describe each change's specifics (code patterns, reasons).
3. **Reference links** — commits, files, PRs, external docs. Only real, accessible links.

### 2. Capture the Session's True Purpose

1. **User's first question/instruction** — the session's real purpose
2. **Topic with the most time spent** — quantitative weight
3. **Final output** — code changes, documents, architecture understanding, etc.

By-products (typo fixes, trivial edits) are not core topics.
However, by-products with commits get a shallow card (2-3 lines + commit link). No commit → omit entirely.

### 3. Card Depth

Not all cards deserve equal depth.

- **Deep card** (decisions + elimination process present) — background, elimination steps, decision table, code, conversation excerpts
- **Shallow card** (simple fixes, config changes) — one-line description + commit link
- **Heuristic**: "If I read this 3 months later, would the code diff alone be enough?" → If yes, keep it shallow. If context is needed, go deep.

### 4. Conversation Excerpts — Capture Turning Points

**The most valuable moments are when the user raises objections, concerns, or pivots direction.**
Extract the 1-2 turns where "Claude proposed A → user rejected/modified → pivoted to B".

```markdown
> **Me:** "Still, not sending events to Amplitude is pretty critical"
> **Claude:** "Since all events go to PubSub, the Amplitude side can be recovered later"
```

If no objections in the session, excerpt the user's key directional statement instead.

### 5. Reference Links

Build commit links from git remote URL + actual hash. **Never fabricate hashes.**

```bash
git -C {path} remote get-url origin
# git@github.com:owner/repo.git → https://github.com/owner/repo/commit/{hash}
```

**Link formatting rules:**
- **No backticks (`) inside link text.** `` [`abc123`](url) `` won't render as a link in markdown.
- **Don't separate links from sentences.** Weave them naturally into the prose.

**Bad:**
```markdown
Ref: [`ecdb7d1`](https://github.com/org/repo/commit/ecdb7d1). File: `src/main.py`
```

**Good:**
```markdown
[Registered an exception handler](https://github.com/org/repo/commit/5d9ccea) to catch uncaught exceptions.
Then [switched to fire-and-forget](https://github.com/org/repo/commit/ecdb7d1) to remove main-thread blocking.
```

### 6. Card Merge/Split Criteria

- **Merge**: Continuation of the same problem (started Monday → finished Wednesday)
- **Split**: Same system but independent problems (macro introduction ≠ full refresh bug)
- Different sessions about the same problem → merge into one card
- Related cards → cross-reference naturally in prose ("see card below", "the schema.yml from the card above is consumed here")

---

## Output Structure

No category grouping. Each `##` is an individual card. Deep cards first, shallow cards last.

```markdown
# NVM Cards ({since_date} ~ {today})

---

## event-hub: Amplitude API stabilization — shared session + circuit breaker + fire-and-forget

**Problem**
(background narrative...)

**Trajectory**
(elimination process, pivots...)

**Resolution**
[Registered an exception handler](https://github.com/org/repo/commit/5d9ccea) to catch uncaught exceptions.
Then [switched to fire-and-forget](https://github.com/org/repo/commit/ecdb7d1) to remove main-thread blocking.
(specific code changes described...)

---

## like: create_event_property idempotent handling

Added [fallback logic](https://github.com/org/repo/commit/cb125cef3) to return existing values instead of errors on duplicate calls.
```

---

## Don'ts

- Chronological summaries
- Meta-commentary ("this was a busy week")
- Links without content
- **Forcing non-decision content into a decision format** ← most common mistake. If it's a learning, write it as a learning. If it's debugging, write it as debugging.
